CASE STUDY 01

Enabling Enterprise-Grade Encryption Without Sacrificing Usability

How I designed a customer-managed encryption key implementation that balanced strict security requirements with seamless user experience for a high-scale SaaS platform

Role

Senior Product Designer

Timeline

6 months

Team

2 Engineering teams(Core & UI), PM, QA, Documentation team

Tools

Figma, Carbon Design System, Dovetail, Mixpanel

Project Context

In the enterprise SaaS landscape, data security isn't just a feature—it's a fundamental requirement that can make or break major deals. This project aimed to implement Bring Your Own Encryption Key (BYOEK) functionality, allowing enterprise clients to maintain control of their encryption keys while using our platform's services.

The Problem

Business Challenge

Enterprise security requirements were blocking major sales opportunities. Our platform's centrally-managed encryption keys, while secure, didn't meet the compliance standards of highly regulated industries (finance, healthcare, government). Prospects required customer-managed encryption keys (BYOEK) to proceed with contracts, but our platform had no native solution.

Key Issues Identified:

  • No customer-managed encryption option for at-rest data
  • Existing encryption solutions had minimal visibility and control
  • Risk that implementing encryption functionality would add complexity and reduce conversion
  • Need to balance seamless UX with stringent security requirements
  • Engineering team concerned about development complexity and timeline
  • Unclear how encryption functionality would integrate into existing product architecture
6 months allocated engineering time requiring strong business case and clear scope

Research & Discovery

I led a comprehensive discovery process to understand enterprise security requirements, competitive landscape, and technical constraints before defining the solution approach.

1. Competitive Analysis & Benchmarking

Analyzed encryption functionality implementations from AWS, Azure, Google Cloud, and other enterprise SaaS platforms. Documented their information architecture, workflow design, configuration complexity, and user feedback patterns.

Key Insights from Research

Trust Through Transparency

Enterprise users needed visibility into encryption status, key management agent health, and data migration progress to trust the system.

Configuration Complexity

Encryption functionality requires technical setup (KMS deployment, JSON configuration) that could overwhelm non-technical stakeholders like compliance officers.

Multi-Persona Requirements

Different roles needed different views: IT teams needed technical details, executives needed high-level status, security teams needed audit logs.

Migration Risk Concerns

Users were extremely worried about data migration from legacy encryption to BYOEK and data loss were primary fears.

Ongoing Monitoring Need

Encryption functionality isn't "set and forget"—users needed continuous health monitoring, status alerts, and diagnostic capabilities.

Support Expectations

Enterprise customers expected white-glove support during BYOEK setup but wanted self-service tools for routine monitoring.

2. User Interviews & Workflow Mapping

Conducted 6 interviews with target customers already in sales conversations who required BYOEK. Focused on understanding: their current key management workflows, technical setup capabilities, migration concerns, and monitoring expectations.

Core Problem Statement

How might we enable enterprise customers to manage their own encryption keys while maintaining platform ease-of-use, minimizing configuration complexity, and providing transparent system monitoring?

The Design Process

3. Information Architecture & System Design

Defined how encryption functionality would integrate into existing product ecosystem. Made critical architectural decisions about entry points, agent registration flows, data migration paths, and system monitoring placement.

Solution 1: Guided Configuration Wizard

The Challenge: Encryption functionality setup required customers to self-deploy a Key Management Agent (KMA) on their infrastructure and configure it via JSON files—intimidating for non-technical stakeholders.

The Solution: Designed a step-by-step wizard with clear validation, examples, and warnings at each stage. Created detailed configuration guides integrated directly into the product documentation platform.

Progressive Disclosure

Broke complex setup into manageable steps with clear progress indicators

Inline Validation

Real-time feedback on configuration errors with specific remediation guidance

Example Templates

Pre-populated JSON configuration templates reducing manual coding errors

Solution 2: Transparent Status Dashboard

The Challenge: Users needed continuous visibility into KMA health, encryption status, and potential issues—but our design system lacked patterns for technical system monitoring.

The Solution: Collaborated with Design Systems team to create reusable UI patterns: status pills, alert banners, connection state indicators, and diagnostic log viewers. Built these as exportable components for future use across products.

  • Real-time KMA connection status with color-coded health indicators
  • Encryption coverage metrics showing which datasets were BYOEK-protected
  • Alert system for configuration issues, connection failures, or key access problems
  • Diagnostic logs accessible to technical users for troubleshooting

Solution 3: Safe Migration Path

The Challenge: Migrating from legacy encryption to BYOEK was users' biggest concern—they needed assurance about data safety and minimal downtime.

The Solution: Designed a migration wizard with clear timeline estimates, rollback capabilities, and validation checkpoints. Included pre-migration testing functionality allowing users to verify configuration before committing to full migration.

Solution 4: Tiered Access Model

The Discovery: Initial plan was to make encryption functionality available to all customers, but this would have complicated the UX for users who didn't need it.

The Solution: After design thinking workshops with PM, proposed restricting encryption functionality to Enterprise/Premium tiers. This allowed us to optimize the core platform for simplicity while providing sophisticated features where needed.

4. Prototyping & Validation

Created interactive Figma prototypes and conducted usability testing with 5 target customers (mix of IT staff and compliance officers). Tested configuration flow, dashboard comprehension, and migration understanding.

Critical Testing Insights

CLI vs. UI Dashboard Debate: Product initially wanted CLI-only interface (faster to build), but testing revealed decision-makers needed visual dashboards for status visibility without SSH access. We deprioritized other engineering tasks to build the UI dashboard first—post-launch, it became our highest-rated feature.

✓ Secured executive buy-in for 6-month engineering investment through clear OKRs

Qualitative Improvements

  • Sales enablement: BYOEK became a key differentiator in competitive enterprise deals, directly contributing to closed contracts
  • Customer confidence: Transparent status monitoring and migration tools reduced anxiety about adopting new security infrastructure
  • Self-service success: Comprehensive documentation and validation reduced dependency on support team during setup
  • Design system evolution: Created foundational patterns for technical system monitoring used across platform
  • Cross-functional alignment: OKR-driven approach unified security, platform, and product teams around shared goals

Key Learnings

This project taught critical lessons about balancing enterprise security requirements with user experience and managing complex stakeholder alignment.

What I Learned

  • Transparency builds trust in security features: Users don't want "black box" security—they want clear visibility into what's happening, especially with encryption. Designing for transparency was more important than hiding complexity
  • User perception equals system trust: Technical implementation matters, but user perception of reliability and control matters equally. Status indicators and diagnostic tools were critical for confidence
  • Stakeholder alignment requires proactive facilitation: Competing priorities (compliance vs. speed, UI vs. API-first, power users vs. simplicity) required documented trade-off thinking and clear "why" explanations
  • Tiered access enables optimization: Restricting complex features to appropriate user segments (Enterprise tier) allowed us to optimize the core platform while serving sophisticated needs where required
  • Testing prevents wrong assumptions: CLI-only approach would have failed with decision-makers. Usability testing caught this early, saving months of post-launch rework

Collaboration & Conflict Resolution

Worked closely with TPM to define deliverables balancing ambition and feasibility for OKRs. Secured executive buy-in by demonstrating clear business impact and phased delivery approach. Facilitated design thinking workshops between Security and Platform Engineering teams with competing priorities, resulting in aligned tiered rollout strategy.

When Product proposed removing customizable peer groups to save development time, I presented research evidence showing this was the most valued feature. Collaborated to identify lower-priority features to descope instead, maintaining core user value while hitting timeline.

Design System Contribution

Created reusable patterns for system status dashboards, agent health monitoring, error codes, and diagnostic logs. These components filled gaps in our design system and accelerated future development across security, monitoring, and compliance features. Documented all patterns in Figma with usage guidelines.
